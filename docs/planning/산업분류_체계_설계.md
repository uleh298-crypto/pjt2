# 산업분류 체계 설계

> What's Your ETF 시스템의 핵심 분류 체계. ETF 클러스터 시각화와 뉴스 영향도 분석에 공통으로 사용됨.

---

## 1. 개요

### 1.1 왜 산업분류 체계가 필요한가?

| 기능 | 산업분류 활용 |
|------|-------------|
| **ETF 클러스터 시각화** | ETF 구성종목을 산업별로 그룹핑하여 버블 차트 표시 |
| **뉴스 영향도 분석** | 뉴스 → 산업 → ETF 2단계 매핑 |
| **포트폴리오 분석** | 사용자 포트폴리오의 산업 분포 차트 |
| **ETF 탐색/필터** | 산업별 ETF 검색 |

### 1.2 설계 원칙

```
표준산업분류(KSIC)를 기반으로 하되,
ETF 투자 관점에서 의미있는 단위로 재그룹핑
```

**핵심 개념:**
- `level`: 표준산업분류 계층 (대→중→소→세분류)
- `group_code`: ETF용 산업 그룹 태그 (레벨 무관)
- `sub_sector`: 테마 ETF용 세분류 코드

## 2. 분류 체계 구조

### 2.1 industry_classification 테이블

```sql
CREATE TABLE "industry_classification" (
    "code" VARCHAR(10) PRIMARY KEY,      -- 산업분류코드
    "name" VARCHAR(100) NOT NULL,        -- 분류명
    "level" INTEGER,                     -- 1=대, 2=중, 3=소, 4=세분류
    "parent_code" VARCHAR(10),           -- 상위분류코드 (셀프참조)
    "group_code" VARCHAR(20),            -- ETF용 그룹 태그
    "group_name" VARCHAR(50),            -- 그룹 한글명
    "created_at" TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.2 level 설명

| level | 설명 | 예시 |
|-------|------|------|
| 1 | 대분류 | C(제조업), J(정보통신업) |
| 2 | 중분류 | 26(전자부품/컴퓨터), 21(의약품) |
| 3 | 소분류 | 032601(반도체 제조업) |
| 4 | 세분류 | SEMI_MEM(메모리 반도체) - 커스텀 |

### 2.3 level과 group_code의 관계

```
┌─────────────────────────────────────────────────────────────────┐
│                    industry_classification                       │
├─────────┬────────┬─────────────────────────┬─────────────────────┤
│ level   │ code   │ name                    │ group_code          │
├─────────┼────────┼─────────────────────────┼─────────────────────┤
│ 1(대)   │ C      │ 제조업                   │ NULL (범위 너무 넓음) │
│ 1(대)   │ J      │ 정보통신업               │ IT_SW               │
│ 2(중)   │ 26     │ 전자부품/컴퓨터...        │ IT_ELEC             │
│ 2(중)   │ 21     │ 의료용 물질 및 의약품     │ BIO                 │
│ 3(소)   │ 032601 │ 반도체 제조업            │ IT_SEMI             │
│ 4(세)   │ SEMI_MEM│ 메모리 반도체           │ IT_SEMI             │
│ 4(세)   │ SEMI_AI │ AI 반도체              │ IT_SEMI             │
└─────────┴────────┴─────────────────────────┴─────────────────────┘
```

**핵심 포인트:**
- `group_code`는 특정 level에 종속되지 않음
- 산업 특성에 따라 적절한 level에서 group_code 부여
- 대분류 C(제조업)는 범위가 너무 넓어서 group_code 없음
- 대분류 J(정보통신업)는 적절해서 바로 IT_SW 부여

## 5. 활용 1: ETF 클러스터 시각화

### 5.0 ETF 구성종목 테이블 구조

ETF 구성종목은 **2개 테이블**에 분산 저장됩니다:

| 테이블 | 저장 데이터 | ETF 유형 | cluster_type |
|--------|------------|----------|--------------|
| `etf_stock_composition` | 주식 기반 구성종목 | sector 있는 ETF | GROUP_CODE/SUB_SECTOR |
| `etf_other_composition` | 선물, 채권, ETF, 우선주 | 파생상품 ETF (12개) | ASSET_TYPE |

**중요**: `etf.sector`가 NULL이어도 `etf_sector_cluster`에 클러스터 데이터가 있어야 합니다.

### 5.1 흐름

**주식 기반 ETF (sector 있음):**
```
etf_stock_composition
       │
       ▼
각 종목의 산업분류(industry_code) 조회
       │
       ▼
industry_code → group_code 매핑
       │
       ▼
group_code별 비중 집계
       │
       ▼
etf_sector_cluster (cluster_type: GROUP_CODE)
```

**파생상품 ETF (sector 없음):**
```
etf_other_composition
       │
       ▼
asset_type별 집계 (FUTURES, BOND, CASH, ETF, PREFERRED_STOCK)
       │
       ▼
etf_sector_cluster (cluster_type: ASSET_TYPE)
```

### 5.2 관련 테이블

```sql
CREATE TABLE "etf_sector_cluster" (
    "id" BIGSERIAL PRIMARY KEY,
    "etf_id" BIGINT,
    "cluster_type" VARCHAR(20) NOT NULL,    -- GROUP_CODE / INDUSTRY / SUB_SECTOR / ASSET_TYPE
    "industry_code" VARCHAR(10),
    "industry_name" VARCHAR(100),
    "group_code" VARCHAR(20),
    "group_name" VARCHAR(50),
    "sub_sector" VARCHAR(100),              -- 테마 ETF용 세분류
    "weight_pct" DECIMAL(6,3) NOT NULL,     -- 비중 (%)
    "stock_count" INTEGER,                  -- 해당 섹터 종목 수
    "pos_x" DECIMAL(10,6),                  -- 버블 X좌표 (UMAP)
    "pos_y" DECIMAL(10,6),                  -- 버블 Y좌표 (UMAP)
    "radius" DECIMAL(10,6),                 -- 버블 반지름
    "distance_to_center" DECIMAL(10,6),     -- ETF 중심까지 거리
    "base_date" DATE NOT NULL,
    "created_at" TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 5.3 산업별 비중 집계 쿼리

```sql
-- ETF의 group_code별 비중 집계
SELECT
    ic.group_code,
    ic.group_name,
    SUM(esc.weight_pct) as total_weight,
    COUNT(DISTINCT s.company_id) as stock_count
FROM etf_stock_composition esc
JOIN stock s ON s.id = esc.stock_id
JOIN company_info ci ON ci.id = s.company_id
JOIN industry_classification ic ON ic.code = ci.industry_code
WHERE esc.etf_id = :etf_id
  AND esc.base_date = :base_date
  AND ic.group_code IS NOT NULL
GROUP BY ic.group_code, ic.group_name
ORDER BY total_weight DESC;
```

## 6. 활용 2: 뉴스-ETF 연결

> ⚠️ **방식 변경 안내** (2025-03-08)
>
> 현재 시스템은 **네이버 증권 종목뉴스** 방식을 사용합니다.
> - 네이버가 이미 뉴스-종목 매핑을 해놓음 (LLM 분석 불필요)
> - `news_stock_mapping` 테이블 사용
> - ETF 관련 뉴스 = ETF 구성종목의 뉴스

### 6.1 현재 흐름

```
[수집] 네이버 증권 종목뉴스
  - finance.naver.com/item/main.naver?code={stockCode}
  - n.news.naver.com에서 본문 추출 (100% 성공)
          ↓
[저장] news_article + news_stock_mapping
  - 뉴스 정보 저장
  - 뉴스-종목 연결 저장
          ↓
[조회] ETF 관련 뉴스
  - etf_stock_composition JOIN news_stock_mapping
  - 구성종목의 뉴스 = ETF 관련 뉴스
```

### 6.2 ETF 관련 뉴스 조회 예시

```
삼성전자 뉴스: "삼성전자, HBM 대규모 수주 성공"
          ↓
news_stock_mapping: 삼성전자 (company_id)
          ↓
┌─────────────────────────────────────┐
│ etf_stock_composition JOIN         │
│ ├─ KODEX 반도체 (삼성전자 35%)       │
│ ├─ TIGER 200 (삼성전자 20%)         │
│ └─ ... (삼성전자 포함 ETF)          │
└─────────────────────────────────────┘
          ↓
해당 ETF 관련 뉴스로 표시
```

### 6.3 관련 테이블

```sql
-- 뉴스-종목 연결
CREATE TABLE "news_stock_mapping" (
    "id" BIGSERIAL PRIMARY KEY,
    "news_id" BIGINT NOT NULL,
    "company_id" BIGINT NOT NULL,
    "created_at" TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT "fk_news_stock_news" FOREIGN KEY ("news_id") REFERENCES "news_article"("id"),
    CONSTRAINT "fk_news_stock_company" FOREIGN KEY ("company_id") REFERENCES "company_info"("id"),
    CONSTRAINT "uk_news_stock" UNIQUE ("news_id", "company_id")
);
```

### 6.4 ETF 관련 뉴스 조회 쿼리

```sql
-- ETF 관련 뉴스 조회 (구성종목 기반)
SELECT na.*, ci.stock_code, ci.company_name, esc.weight_pct
FROM news_article na
JOIN news_stock_mapping nsm ON na.id = nsm.news_id
JOIN company_info ci ON nsm.company_id = ci.id
JOIN etf_stock_composition esc ON ci.id = esc.company_id
WHERE esc.etf_id = :etf_id
ORDER BY esc.weight_pct DESC, na.published_at DESC
LIMIT 20;
```

### 6.5 장점

| 항목 | 설명 |
|------|------|
| **100% 본문 추출** | 네이버 뉴스 형식 통일 |
| **LLM 비용 절감** | 네이버가 이미 뉴스-종목 매핑 해놓음 |
| **정확한 매핑** | 네이버 자체 분석 기반 (사람이 검수) |
| **간단한 구조** | 종목뉴스 → ETF 관련 뉴스는 JOIN으로 조회 |

## 9. 주의사항

### 9.1 group_code 할당 원칙

1. **가장 구체적인 level에서 할당**: 세분류가 있으면 세분류에, 없으면 소분류에 할당
2. **범위가 넓은 대분류는 NULL**: C(제조업)처럼 범위가 너무 넓으면 group_code 없음
3. **하위 분류는 상위의 group_code 상속 가능**: SEMI_MEM(level 4) → IT_SEMI (032601에서 상속)

### 9.2 새로운 산업/테마 추가 시

1. 기존 체계에 맞는 level 결정
2. 적절한 parent_code 연결
3. group_code 할당 (기존 그룹 or 신규 그룹)
4. etf_sector_cluster 재계산 배치 실행

### 9.3 테마 ETF 처리

테마 ETF(예: AI ETF, 메타버스 ETF)는 여러 산업에 걸쳐 있을 수 있음:

```
AI ETF 구성:
├─ IT_SEMI (AI 반도체) - 40%
├─ IT_SW (AI 소프트웨어) - 35%
├─ IT_ELEC (AI 서버) - 15%
└─ TELECOM (AI 인프라) - 10%
```

이 경우 `etf_sector_cluster`에 여러 group_code로 분리 저장됨.
